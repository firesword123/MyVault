# MyVault Note 模块架构评估摘要

## 1. 模块定位

Note 模块是当前 MyVault 的核心业务模块，承担本地笔记的创建、编辑、保存、归档、软删除、恢复、移动与文件夹组织。  
它不是数据库型笔记系统，而是一个**文件系统驱动**的本地笔记工作台，设计上明确借用了 Windows 文件管理语义：

- 笔记对应真实文件
- 文件夹对应真实目录
- `drafts / inbox / trash` 是显式系统目录
- UI 状态最终应服从磁盘状态

这意味着 Note 模块的架构评估重点，不只是前端交互，还包括：

- 文件系统一致性
- 笔记状态迁移语义
- 目录操作与路径安全
- 文本编码稳定性
- UI 与磁盘状态是否可能失配

## 2. 当前技术栈

### 前端

- React 19
- TypeScript
- Vite
- `@tauri-apps/api`
- `@tauri-apps/plugin-opener`
- `lucide-react`

### 桌面容器

- Tauri 2

### 后端

- Rust
- `serde`
- `serde_json`

### 当前实现方式

- 前端负责交互、局部状态、编辑器渲染与调用 Tauri command
- Rust 后端负责：
  - 数据目录初始化
  - 文件/目录读写
  - 路径校验
  - 笔记元数据序列化
  - 软删除/恢复/移动
  - 工作区扫描与返回

---

## 3. 模块边界与职责

### 前端职责

- 左栏文件夹与笔记列表展示
- 搜索、选中、拖拽、当前目录高亮
- 笔记编辑区与格式化工具
- 保存、删除、恢复、彻底删除、移动等入口
- 将操作翻译为 Tauri command
- 展示后端错误信息与确认提示

### 后端职责

- 维护 `myvault-data/vault/notes` 下的数据结构
- 统一处理笔记文件和目录树
- 确保系统目录存在：
  - `drafts`
  - `inbox`
  - `trash`
- 负责笔记 frontmatter 的读写
- 负责合法路径校验与系统目录保护
- 负责真实文件移动，而不是仅更新内存状态

---

## 4. 当前数据结构

### 目录结构

```text
myvault-data/
  settings.json
  vault/
    notes/
      drafts/
      inbox/
      trash/
      <用户自定义目录...>
```

### 笔记文件格式

笔记以 UTF-8 文本保存，包含 frontmatter 与正文：

```md
---
id: note-123
title: Example
created_at: 1710000000
updated_at: 1710000100
original_folder_path: projects/demo
deleted_at: 1710000200
---
<p>Hello</p>
```

### 字段语义

- `id`
  - 稳定笔记标识
  - 当前也用于文件名
- `title`
  - 展示标题
- `created_at`
  - 创建时间
- `updated_at`
  - 最近更新时间
- `original_folder_path`
  - 回收站恢复时优先返回的目录
- `deleted_at`
  - 软删除时间
- 正文
  - 当前以 HTML 兼容内容存储

## 5. 当前已实现功能

- 新建笔记

- 切换笔记前自动落盘草稿

- 搜索标题 / 摘要 / 路径 / 文件夹
- 将当前笔记移动到任意现有文件夹

- 正文编辑

- 打开当前笔记所在文件夹

## 6. 当前交互体验特征

从体验角度，当前 Note 模块已经具备“单人本地笔记工具”的完整基础闭环：

- 左栏是典型文件夹/笔记树形工作流
- 中间是沉浸式编辑区
- 回收站替代直接删除，降低误删风险
- 文件系统能力与 UI 操作保持一致
- 文件夹级操作和笔记级操作已经有基本保护

用户能完成的典型流包括：

1. 新建笔记
2. 编辑与自动保存
3. 拖到某个目录整理
4. 删除到回收站
5. 从回收站恢复或彻底删除

从可用性上看，这已经不是 demo 级别，而是进入了“可长期演进的核心模块”状态。

## 7. 当前架构优点

### 7.1 数据透明

数据直接落盘，结构清晰，便于：

- 用户理解
- 手工备份
- 外部工具接入
- 问题排查

### 7.2 业务语义直接映射到文件系统

相比引入数据库或抽象存储层，当前实现更直观：

- 移动笔记 = 移动文件
- 删除到回收站 = 移动到 `trash`
- 文件夹重命名 = 重命名目录

这种映射对架构理解和维护成本是有利的。

### 7.3 后端边界清晰

Rust 后端没有做复杂领域建模，但已经承担了最关键的稳定性职责：

- 真实读写
- 路径校验
- 状态迁移
- 系统目录保护

这是当前结构里最稳的一层。

### 7.4 前后端职责分离还算健康

前端没有直接操作文件系统，所有真正的数据变更都经过 command。  
这使得未来抽出 service 层、做测试或替换 UI 都还有空间。

### 7.5 已补基本自动化验证

目前已有后端测试与 smoke 检查，至少对以下点形成了基本保护：

- UTF-8 读写
- frontmatter roundtrip
- 路径校验
- 回收站/恢复/移动/空目录树删除流转

---

## 8. 当前架构不足与评估重点

### 8.1 编辑器实现偏轻量，长期扩展风险高

当前编辑区基于：

- `contentEditable`
- `document.execCommand`

这套方案可以快速工作，但在长期演进上存在明显风险：

- 浏览器行为不稳定
- 粘贴与撤销语义难控
- 复杂格式化能力难扩展
- HTML 结构容易随交互逐步变脏

如果未来要扩展：

- Markdown 结构化编辑
- 引用/列表/代码块高级操作
- 更稳定的撤销重做
- 块级编辑

那么当前编辑器实现会成为瓶颈。

### 8.2 领域模型仍偏“文件工具层”，而不是“笔记领域层”

现在后端更像一层文件操作服务，而不是完整的 Note Domain Service。  
例如：

- 笔记状态没有显式枚举模型
- 系统目录语义散落在多个函数中
- 操作返回主要是结构体，不是更清晰的领域事件

现阶段够用，但模块继续变大后，复杂度会开始外溢。

### 8.3 前端状态仍有一定分散

`App.tsx` 目前承担了较多职责：

- 工作区加载
- 笔记切换
- 错误映射
- 草稿落盘
- 目录前缀重写
- 删除/恢复/移动/空回收站

这意味着：

- 状态协调点较多
- 测试不够容易
- 后续功能继续增加时，组件层会继续变重

### 8.4 搜索与工作区刷新仍是全量扫描思路

当前工作区刷新依赖扫描目录树并返回完整结果。  
对当前体量足够，但当笔记量增长后可能遇到：

- 启动扫描成本上升
- 删除/恢复/移动后的全量刷新偏重
- 搜索全部依赖前端内存过滤

如果未来目标是大规模笔记仓库，这一层需要重新评估。

### 8.5 文件格式仍是“HTML 正文 + frontmatter”

这种格式的优点是简单、直接；缺点是：

- 不完全是标准 Markdown 工作流
- 外部编辑器兼容性一般
- HTML 内容长期维护成本较高
- 若未来需要结构化导出/转换，处理会更复杂

---

## 9. 当前适合的产品阶段判断

以当前架构看，Note 模块适合以下阶段：

- 单机本地优先
- 笔记量中小规模
- 个人知识整理 / 轻量项目笔记
- 明确依赖真实文件系统而非数据库

当前还不适合直接承载以下方向，至少要先重构部分架构：

- 多人协作
- 云同步冲突解决
- 高复杂富文本编辑
- 大规模知识库全文检索
- 插件生态

---

## 10. 建议的架构评估结论

### 可以给出的正向结论

- 当前 Note 模块已经具备完整核心闭环
- 文件系统驱动模型清晰，利于理解和排障
- 数据可见、可迁移、可人工恢复
- 后端对关键数据路径已有一定保护
- 作为本地优先桌面笔记工具，架构方向是成立的

### 需要重点追问的问题

1. 中长期是否继续坚持“文件系统优先”而不引入数据库索引层？
2. 编辑器是否会升级为更强的结构化编辑方案？
3. 后端是否需要抽出更明确的 Note/Folder/Trash 领域服务层？
4. 工作区刷新是否需要增量化或索引化？
5. 是否会引入同步、历史版本、冲突处理等更复杂能力？

### 如果继续演进，推荐优先级

1. 抽出 Note service / Folder service / Trash service
2. 给前端状态流做更清晰的 action 化或 store 化
3. 评估替换 `contentEditable + execCommand`
4. 设计增量索引或缓存层
5. 明确文件格式长期策略：HTML、Markdown、或混合结构

---

## 11. 适合带去评估会的简版结论

MyVault 当前的 Note 模块，本质上是一个**以真实文件系统为存储真相、由 Rust 后端负责文件语义、由 React 前端负责交互与编辑体验的本地优先笔记模块**。  
它的优势是结构直观、数据透明、与 Windows 文件管理语义一致；它的主要风险点则集中在编辑器实现、前端状态复杂度、以及未来规模化后的全量扫描与格式演进能力。

如果评估目标是“这套架构能不能支撑当前阶段继续做”，答案是：**可以**。  
如果评估目标是“这套架构能不能直接承载更复杂的编辑、同步和大规模知识库”，答案是：**需要先做一轮有意识的架构升级**。
